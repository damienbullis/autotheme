# Design Gaps & Missing Capabilities

Structural holes in the current design that prevent autotheme from serving its intended purpose.

---

## GAP-1: No concept of "layers" in the output

**Problem:** AutoTheme outputs a single flat list of CSS variables. There's no structural separation between:

- Raw palette values (atomic building blocks)
- Semantic mappings (what role each color plays)
- Framework bindings (Shadcn-specific names)
- System scales (typography, spacing)

**Impact:** Consumers can't import just the palette, or just the semantics. It's all or nothing.

**What's needed:** Output should be structured in clearly delineated sections (comments are fine) or optionally split into separate files. Each layer should be independently toggleable.

---

## GAP-2: Semantic tokens don't reference palette variables

**Problem:** Semantic tokens (surface, primary, accent, etc.) are computed independently from the palette. They don't use `var(--color-primary-500)` or similar references. They're standalone OKLCH values.

**Impact:**

- Changing a palette value doesn't cascade to semantic tokens
- The semantic layer and palette layer can disagree (different colors for "primary")
- No way to understand the relationship between `--primary` and `--color-primary-500`

**What's needed:** Semantic tokens should either:

1. Reference palette variables directly: `--primary: var(--color-primary-500)`
2. Or be computed from the same palette entries (and documented which)

Option 1 is ideal for composability - change the palette, semantics update automatically. For a one-time generation tool, option 2 is acceptable if well-documented.

---

## GAP-3: No design system rules engine

**Problem:** The mapping from palette to semantic roles is hardcoded in `semantic.ts`. There's no way for consumers to say "I want surface-dim to use the 900 shade" or "I want borders to use the 700 shade with 20% opacity."

**What's needed:** A configurable mapping layer:

```typescript
semantics: {
  rules: {
    surface: { source: "primary", shade: 950 },
    surfaceElevated: { source: "primary", shade: 900 },
    border: { source: "primary", shade: 700, alpha: 0.2 },
    accent: { source: "tertiary", shade: 500 },
  }
}
```

This turns the semantic layer from a black box into a transparent, tweakable system.

---

## GAP-4: No concept of color roles beyond 4

**Problem:** AutoTheme names harmony colors primary/secondary/tertiary/quaternary. Beyond 4 colors, there are no semantic names. Custom harmonies can generate any number of colors, but only the first 4 get names.

**What's needed:** Either:

- Allow custom naming for harmony colors
- Or a clear convention for colors beyond 4 (quinary, senary, or indexed)

---

## GAP-5: No surface/elevation system

**Problem:** Modern design systems (Material, Apple HIG, IBM Carbon) define elevation as a key concept. Surfaces at different elevations have different visual properties (lighter/darker, more/less shadow). AutoTheme has no elevation model.

**What's needed:** An elevation system that coordinates:

- Surface lightness at each elevation level
- Shadow depth at each elevation level
- Border treatment at each elevation level

This doesn't need to be complex. Even 3-5 elevation levels with coordinated surface + shadow values would be transformative.

---

## GAP-6: No interactive state tokens

**Problem:** Design systems need hover, active, focus, and disabled states. AutoTheme generates no state variants. Every consumer must manually compute:

```css
--button-hover: /* ??? lightened primary? darkened? */ --input-focus:
  /* ??? ring color from primary? */
  --card-active: /* ??? */;
```

**What's needed:** State modifiers for semantic colors:

```css
--primary-hover: oklch(/* primary lightened/darkened */);
--primary-active: oklch(/* primary further adjusted */);
--primary-disabled: oklch(/* primary desaturated, lower contrast */);
```

These follow predictable patterns (hover = +/- 5% lightness, active = +/- 10%, disabled = 50% saturation + reduced alpha) and could be auto-generated.

---

## GAP-7: No color swatch documentation in output

**Problem:** The generated CSS has no comments explaining what each variable represents or how to use it. For a one-time generation tool where users are expected to modify the output, documentation in the output is critical.

**What's needed:** Optional comments in generated CSS:

```css
/* Surface: Main background - uses primary-950 */
--surface: oklch(0.12 0.01 250);

/* Text hierarchy: 5 levels from prominent to muted */
--text-1: oklch(0.93 0.01 250); /* Primary text */
--text-2: oklch(0.85 0.015 250); /* Emphasis */
```

Toggle: `comments: true/false` in config.

---

## GAP-8: Palette doesn't handle perceptual uniformity

**Problem:** Tints and shades are generated by linear HSL lightness steps (+10%, -10%). This produces visually uneven steps because HSL lightness is not perceptually uniform. A 10% lightness change at L=90 looks different from 10% at L=50.

**What's needed:** Option to generate variations using OKLCH lightness (which IS perceptually uniform). The Color class already supports OKLCH conversion - the variation generation should offer a `perceptual: true` mode that operates in OKLCH space.

---

## GAP-9: No theme metadata in output

**Problem:** Generated CSS doesn't include any metadata about how it was generated. If you open a file months later, you don't know what color, harmony, or config produced it.

**What's needed:** A metadata comment block at the top:

```css
/* Generated by AutoTheme v0.1.0
 * Primary: #7aa2f7
 * Harmony: square
 * Mode: dark
 * Generated: 2026-03-01
 */
```

---

## GAP-10: No way to extend with custom token categories

**Problem:** AutoTheme has a fixed set of output categories. If a consumer needs something custom (e.g., animation timing tokens, z-index scale, opacity scale), there's no extension mechanism.

**This is a lower priority gap.** For now, consumers can append custom tokens to the generated file. But a `custom` config section could allow arbitrary key-value pairs to be included in the output.
